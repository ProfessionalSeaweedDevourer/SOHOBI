PR #$ARGUMENTS 를 리뷰하라.

**인자**: `$ARGUMENTS` — 첫 번째 공백 구분 토큰을 PR 번호로 사용한다. 나머지는 컨텍스트 메모로 무시한다.

1. `gh pr view <PR번호>`로 PR 정보(제목, 본문, 변경 파일) 확인
2. `gh pr diff <PR번호>`로 전체 diff 확인
3. **변경 파일 성격에 따라 리뷰어 페르소나 자동 선택** (아래 라우팅 표 기준). 선택된 페르소나의 관점을 리뷰 프롬프트에 적용한다. 복수 조건 매치 시 우선순위 순으로 한 개만 선택:

   | 우선순위 | 변경 파일 조건 | 페르소나 |
   | ---- | -------------- | -------- |
   | 1 | `backend/signoff/`, `backend/auth_router.py`, `.env*`, `.gitignore`에 민감 패턴 추가/삭제 | **시니어 보안 엔지니어** — 위협 모델·누설 경로·권한 경계·회귀 봉인 구조를 의심부터 시작 |
   | 2 | `backend/db/`, SQL 마이그레이션, 스키마 변경 | **시니어 DB 엔지니어** — 인덱스·락·마이그레이션 가역성·데이터 정합성 |
   | 3 | `backend/agents/`, `backend/orchestrator.py`, `backend/api_server.py` 중심 | **시니어 백엔드 엔지니어** — 에이전트 계약·타임아웃·레이턴시·실패 모드 |
   | 4 | 프론트엔드 파일(`frontend/`) 비중 >50% | **시니어 프론트엔드 엔지니어** — 접근성·렌더 성능·상태 누수·번들 영향 |
   | 5 | 위 모두 해당 없음 (문서·설정·테스트 단독 등) | **기본** — 페르소나 삽입 없이 CLAUDE.md 기조(회의적·검증 가능한 사실) 유지 |

   선택된 페르소나는 리뷰 코멘트 상단에 명시하지 않아도 되며, 평가 관점에만 반영한다. 과도한 위협 상상·있지 않은 취약점 언급은 금지.

4. 변경된 소스 파일을 읽고 다음을 점검:
   - 코드 품질, 버그, 보안 취약점
   - DOMAIN=localhost 회귀 여부 (grep 필수)
   - .env 파일 커밋 여부
   - CLAUDE.md의 프로젝트 규칙 준수 여부
5. 실행 가능한 테스트가 있으면 실행
6. 리뷰 결과를 PR에 코멘트로 남긴다 (`gh pr comment <PR번호>`)
