세션 인수인계 블록을 읽고 이번 세션에서 처리 가능한 분량의 작업을 선정·기획하라.

전제: 이미 `/boot`로 컨텍스트가 복원되었거나, 본 커맨드 실행 중 자체 복원을 포함한다. 인자는 받지 않는다.

1. 참조할 handoff 파일을 결정한다 (우선순위 순):
   a. **IDE에서 현재 열린 파일**이 `docs/session-reports/*-handoff.md` 패턴이면 그 파일을 사용한다 (ide_selection 컨텍스트 또는 활성 에디터 탭)
   b. 그렇지 않으면 `docs/session-reports/`에서 `ls -t | head -1`로 최신 파일을 선택한다
   - 사용자가 특정 handoff를 열어두고 `/specify`를 호출한 경우, 그것이 이 세션에서 다루고 싶은 대상이라는 신호로 간주
   - `CLAUDE_HANDOFF_START` ~ `CLAUDE_HANDOFF_END` 블록만 읽는다 (인간용 산문 전체 읽기 금지)
   - 이미 `/boot`로 로드된 경우 재읽기 생략

2. 세션 크기 후보를 분류한다:
   - 작업 성격별: 조사/검증, 단일 파일 코드 수정, 다중 파일 리팩토링, UI 작업
   - 의존 관계: `next` 필드의 순서를 존중하고, 선행 항목이 미완이면 후속은 후보에서 제외
   - 블로커: `traps`·`decisions`에 명시된 제약(모델 분화, 로컬/프로덕션 불일치 등)을 반드시 검토

3. 2~3개의 세션-사이즈 후보를 사용자에게 AskUserQuestion으로 제시하고 하나를 선택받는다
   - 각 후보에 예상 범위, 코드 변경 여부, 차단 요인을 1줄씩 표기
   - 세션 1회로 완결 불가능한 스코프(예: 전체 재설계 A/B/C)는 쪼개서 제시

4. 선택된 범위를 Plan mode 플랜 파일로 구체화한다:
   - Context (왜 이 작업인가)
   - 변경할 파일 경로
   - 재사용 가능한 기존 함수/유틸 (Grep으로 실제 확인)
   - 검증 절차 (curl / pytest / playwright 구분)
   - 세션 외로 밀어낼 항목 명시

5. 플랜을 보고하고 ExitPlanMode로 승인을 요청한다

주의:
- handoff의 HIGH라도 프로덕션 검증 환경이 필요한 항목은 사용자에게 기동 상태를 먼저 확인
- 본 커맨드는 Plan mode 진입 상태에서 호출되는 것을 전제로 한다. 아닐 경우 계획만 텍스트로 제시하고 코드 변경은 하지 않는다
