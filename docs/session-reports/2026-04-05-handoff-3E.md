## ⚠️ 투표 시스템 보안 취약점 (미해결)

현재 `POST /api/roadmap/vote`는 **session_id만으로 중복 투표를 방지**하고 있어 다음과 같은 우회가 가능합니다.

### 취약점 목록

| 수준 | 취약점 | 우회 방법 |
|------|--------|-----------|
| 높음 | session_id 위조 | 브라우저 DevTools → localStorage `sohobi_session_id` 임의 값으로 변경 후 재투표 |
| 높음 | 새 세션 반복 생성 | AI 에이전트와 새 대화 시작 → 새 session_id 발급 → 무한 투표 가능 |
| 중간 | 서버 측 session_id 검증 없음 | `POST /api/roadmap/vote`에서 session_id가 실제 session_store에 존재하는지 확인 안 함 |
| 중간 | Rate limiting 없음 | 동일 IP에서 단시간 대량 요청 가능 |

### 개선 방안 (우선순위 순)

1. **session_id 유효성 검증** (즉시 적용 가능, 백엔드만 수정)
   - `toggle_vote()`에서 `session_store.get_session(sid)` 호출 → None이면 400 반환
   - 이미 `session_store.py`가 있으므로 추가 의존성 없음

2. **JWT 로그인 사용자만 투표 허용** (auth_router.py 이미 구현됨)
   - `toggle_vote()`에 `get_optional_user` 의존성 추가
   - JWT 없으면 투표 거부 (`401` 반환)
   - 프론트엔드: 로그인 상태일 때만 투표 버튼 활성화

3. **1인 1기능 1표 제한 강화**
   - JWT 사용자 기준으로 `vote_{feature_id}_{user_id}` 로 문서 ID 변경
   - 현재 `vote_{feature_id}_{session_id}` → 로그인 시 `user_id` 기반으로 전환

4. **Rate limiting** (추후)
   - `slowapi` 라이브러리 또는 Azure API Management 레이어에서 IP당 분당 요청 제한
