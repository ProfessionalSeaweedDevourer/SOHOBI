# 검색 엔진 자체 호스팅 전환 — 쉬운 설명

> 본 문서는 [strategic/self-hosted-rag-and-db-cost-optimization.md](strategic/self-hosted-rag-and-db-cost-optimization.md)의 결론을 기술 비전공자도 이해할 수 있도록 풀어쓴 안내문이다. 결정 근거의 정량 분석이 필요하면 strategic 문서를 우선 참고.

## 1. 무엇이 바뀌나 — 한 줄 요약

**Azure에 매월 ₩100,000씩 빌리던 검색 서비스를 끊고, 우리가 이미 빌리고 있는 데이터베이스 안에 검색 기능을 직접 심는다.**

사용자(웹사이트 방문자) 입장에서는 보이는 변화가 없다. 검색 결과·응답 속도·기능 모두 동일하게 유지하는 것이 목표.

## 2. 왜 바뀌나

### 검색은 어떻게 작동하는가 (배경)

SOHOBI 백엔드는 사용자가 "강남역 카페 창업 시 위생법 신경 써야 할 부분?" 같은 질문을 하면 다음 순서로 답한다:

1. 질문을 의미를 담은 숫자 묶음(**임베딩 벡터**, 약 1,500~3,000개의 실수)으로 변환
2. 미리 만들어 둔 **법령 문서 벡터 DB**에서 의미가 가장 가까운 법령 조항 3-5개 찾기
3. 찾은 조항을 LLM(gpt-5.4-mini)에게 컨텍스트로 주고 답변 생성

여기서 2번 단계의 "벡터 DB"가 지금까지 **Azure AI Search**라는 외부 서비스였다.

### 비용 구조의 함정

Azure AI Search Basic 등급은:
- **데이터를 0 byte 저장해도, 쿼리를 0건 보내도 매월 ~$73 (₩100,000+) 정액**
- 마치 짐을 하나도 안 둬도 매월 임대료가 나오는 창고

지금 SOHOBI는:
- 실제 사용자 트래픽이 거의 없는 portfolio 단계
- 라이브 서버에서도 AI Search 연결이 차단된 상태로 정상 운영 중 (즉, 이미 한 달 넘게 돈만 내고 안 쓰고 있었음)

→ 이 정액 임대료를 **즉시 끊는** 결정이 PR #339였다.

### 그러면 검색 기능은?

검색 기능 자체는 사용자에게 필요하므로, **돈 안 드는 다른 방법**으로 같은 기능을 구현해야 한다. 후보 4개를 비교해 본 결과, 우리가 이미 사용 중인 **PostgreSQL 데이터베이스**에 검색 확장 기능을 추가하는 방법이 압도적으로 유리했다.

## 3. 대안 4가지를 어떻게 비교했나

다소 단순화한 비유:

| 후보 | 비유 | 장점 | 단점 |
|------|------|------|------|
| **PostgreSQL pgvector** | 이미 빌린 창고 한 칸을 비워서 검색용으로 쓰기 | 추가 임대료 0, 운영 도구 그대로 | (사실상 단점 없음) |
| SQLite + sqlite-vec | 노트북 가방 안에 작은 메모지로 검색 만들기 | 간단함 | 컨테이너 재시작 시 사라짐 |
| FAISS | 외워서 즉답하는 방식 | 빠름 | 매번 책상에 앉을 때마다 다시 외워야 함(cold start) |
| ChromaDB | 새 도구 사오기 | Python 친화적 | 또 다른 운영 대상 추가 |

**결정**: PostgreSQL pgvector. 핵심은 "이미 운영 중인 자원에 기능 하나를 더 얹는다"는 점.

## 4. 무엇을 해야 하나 (개발 작업)

5단계로 나눠서 진행한다:

### Phase A — 그릇 만들기 (회귀 위험 0)
백엔드 코드에서 "검색 엔진"을 호출하는 부분을 **추상 인터페이스**로 분리한다. 지금까지는 코드가 직접 Azure AI Search SDK를 부르고 있었지만, 이걸 `검색기.질의()` 같은 일반화된 호출로 바꾸고 실제 구현체는 갈아끼울 수 있게 한다.

→ 이 단계만 해도 동작은 그대로다. 회귀 0건 보장.

### Phase B — pgvector 활성화 + 데이터 적재
PostgreSQL에 명령어 한 줄(`CREATE EXTENSION vector;`)로 벡터 검색 확장을 켠다. 그리고 법령 문서를 임베딩으로 변환해서 PG에 적재하는 스크립트를 만든다.

```
JSONL 파일 (4,500개 법령 조항 + 임베딩) → PostgreSQL legal_docs 테이블
```

### Phase C — pgvector 구현체 + 듀얼 운영
Phase A에서 만든 추상 인터페이스의 두 번째 구현체로 pgvector 버전을 만든다. 환경변수 `RAG_BACKEND=pgvector`로 전환 가능하게.

이 시점에 **두 검색기를 동시에 운영하면서 같은 질문에 누가 더 나은 답을 주는지** 비교 측정한다 (recall@5, latency).

### Phase D — 정부지원사업 인덱스
법령 외에 정부지원사업 인덱스도 동일한 방식으로 옮긴다. 단 이건 외부 의존(데이터 소스 미확정)이 풀려야 진행 가능.

### Phase E — Azure AI Search 모듈 영구 폐기
Phase C가 검증된 후 Bicep 모듈을 코드에서 완전 삭제. 더 이상 누구도 실수로 재활성화하지 못하게.

## 5. 시간·돈 측면 효과

| 항목 | 변경 전 | 변경 후 |
|------|--------|--------|
| 월 청구액 (총) | ~$95-110 (₩133,000-154,000) | ~$22-37 (₩31,000-52,000) |
| 검색 응답 latency P95 | ~300ms | ~500ms 목표 (200ms 증가 허용) |
| 인프라 자원 수 | 9개 | 8개 (AI Search 제거) |
| cutover 백업 채널 | 3개 (Cosmos·Blob·PG) | 2개 (Cosmos·PG로 통합) |

## 6. 리스크 — 솔직히

자체 호스팅의 약점:
- **PostgreSQL 한 곳에 모든 데이터가 모인다**. PG가 죽으면 검색 + 상권 데이터 동시에 영향. portfolio 단계라 SLA 요구 0이라서 수용 가능하지만, 실 사용자 트래픽 발생 시 read replica 등 보강 필요.
- **시맨틱 리랭킹 품질**. Azure AI Search는 자체 리랭커 모델이 있어서 검색 결과 정렬이 똑똑하다. pgvector는 단순 코사인 유사도라 리랭커 자체 구현이 필요할 수 있음. 이게 Phase C 품질 비교의 핵심 변수.

## 7. 사용자가 결정해야 하는 것

기술팀 합의 필요 사항:
1. **이 안(pgvector)에 동의하는가?** — 동의 시 Phase A 즉시 착수
2. **언제 시작할까?** — 즉시 vs 다른 작업 마무리 후
3. **검색 품질 평가 정답 셋을 누가 만들까?** — 도메인 지식 필요(법령·정부지원사업 분야)

기술팀 외부 작업:
4. **정부지원사업 원본 데이터 확보** — Phase D의 외부 블로커. 데이터 보유자 식별 + 접근 권한 확보가 별도 워크스트림.
