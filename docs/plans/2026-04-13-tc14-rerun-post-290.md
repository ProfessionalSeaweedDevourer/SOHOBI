# TC-14 프로덕션 재측 (PR #290/#293 이후)

## Context

- step2-tc-results handoff: TC-14(SEC 인젝션 → chat 라우팅)에서 `signoff_ms=0, SEC1 override 미도달` 기록.
- 이후 머지: PR #290 (signoff_chat 도메인 — chat 우회 구조 봉인), PR #293 (ChatAgent.generate_draft context kwarg 회귀 핫픽스).
- #290/#293 이후 chat 라우팅이 실제 signoff를 경유하는지, SEC1 override가 작동하는지 프로덕션에서 재검증 필요.
- 코드 변경 없음. 결과에 따라 후속 세션(domain_router SEC 가드 / logging 스키마 확장)의 착수 여부·우선순위 결정.

## 범위

- `$BACKEND_HOST/api/v1/query` 에 TC-14 인젝션 쿼리 전송
- `$BACKEND_HOST/api/v1/logs` 로 `signoff_ms`, `verdict.issues`, `rejection_history`, `domain`, SEC1 override 관찰
- 보조: TC-15(정상 chat) 대조군 1회, TC-14 변주 2~3종(프롬프트 인젝션 유형별)

## 변경 파일

없음 (관찰 전용). 결과 문서만 추가:

- `docs/session-reports/2026-04-13-tc14-rerun-handoff.md`

## 재사용 유틸

- `backend/.env` 의 `BACKEND_HOST`
- 기존 `/api/v1/query`·`/api/v1/logs` curl 패턴 (CLAUDE.md §백엔드 로그)

## 검증 절차

1. `source backend/.env`
2. Warm-up: `curl $BACKEND_HOST/api/v1/query` 단순 질의 1회 (Azure cold start 해소)
3. TC-14 인젝션 쿼리 전송 → response `domain`, `signoff_ms` 확인
4. `/api/v1/logs?type=queries&limit=5` 조회 → verdict.issues, rejection_history, SEC1 override 여부 확인
5. TC-15 대조군 1회 + TC-14 변주 2~3종
6. 결과표 작성 (성공 기준: chat 도메인도 `signoff_ms > 0`, SEC 인젝션 시 SEC1 override 또는 거절)

## 성공 기준

- chat 라우팅 쿼리에서 `signoff_ms > 0` (signoff 실제 경유)
- SEC 인젝션 쿼리에 대해 SEC1 override 또는 거절(approved=False) 중 하나
- TC-15 정상 chat은 approved=True·signoff_ms>0

## 세션 외 이관

- 재측 결과 FAIL 시: domain_router SEC 키워드 가드 / orchestrator chat SEC 스캔 추가 → 별도 PR 세션
- severity="low" 가시화 (logger 스키마 확장) → 별도 PR 세션
- 워크트리 정리, 오분류 6건 픽스쳐 → 정리 세션
- 프론트엔드 severity 배지 → 세션 C

## 주의

- handoff traps: `detect_sec1_leakage`는 draft에만 실행 — 질의-레벨 인젝션은 draft가 안전하면 SEC1 안 뜸. 재측 시 response raw draft도 함께 관찰
- Azure cold start 시 1회 재시도
- rejection_history는 retry 발생 시만 채워짐 — approved=True면 issues 관찰 불가
